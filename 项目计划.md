## 个性化旅游系统项目计划（整合版）

说明：本文档用于课程阶段性提交与开发执行，已将新增需求整合进统一结构，后续按里程碑滚动更新。

---

## 1. 项目目标与范围

### 1.1 项目目标
构建一个前后端分离的个性化旅游系统，面向“查询-规划-推荐-记录-展示”全流程场景，支持从地点搜索到路线生成再到内容沉淀的闭环。

### 1.2 范围定义
1. 前端：Vue 3，负责交互输入、地图展示、结果可视化。
2. 后端：FastAPI，负责业务编排、算法调用、接口输出。
3. 数据层：SQLite + SQLAlchemy，负责 POI、设施、用户、日记等数据持久化。
4. 核心算法：必须手写（禁止核心模块调用 networkx 等高级算法库）。

### 1.3 技术红线
1. 核心算法手写：Graph/Dijkstra、Trie、Heap、Huffman、倒排索引。
2. 查询主路径不得退化为 O(n) 全表扫描。
3. 距离排序以图路径距离为准，不以直线距离直接替代。
4. 接口字段变更必须同步更新接口文档。

## 2. 系统架构与目录职责

### 2.1 总体架构
1. API 层：路由定义、参数校验、统一响应。
2. Service 层：业务编排，连接 DB 与算法模块。
3. Core 层：数据结构与核心算法实现。
4. DB 层：模型定义、会话管理、CRUD。

### 2.2 核心目录职责（后端）
1. app/api：接口入口与协议层。
2. app/services：业务流程编排与策略组合。
3. app/core：手写算法与数据结构。
4. app/db：数据库模型、连接与数据访问。
5. app/models：Pydantic 请求/响应模型。

## 3. 五大功能模块与关键考查点

### 模块一：旅游场所查询与管理（数据基础）
目标：建立可扩展的数据底座与高效查询能力。

关键点：
1. 海量数据支持：
   - 200+ 景区/校园。
   - 每个景区 20+ 建筑物。
   - 10+ 类设施、总数 50+。
2. 数据模型：
   - 设计 poi_table、facility_table、poi_facility_relation（可扩展用户关联表）。
3. 高效查询：
   - 多关键字与模糊查询。
   - 系统启动后构建 Trie 索引，支持前缀与拆词检索。
4. 距离排序：
   - 查询结果按真实路径长度排序。
   - Top-K 采用手写堆，避免全量排序。

### 模块二：智能路线规划（核心算法）
目标：提供可解释、可扩展的路径规划能力。

关键点：
1. 地图可视化：后端返回路径点序列，前端地图绘制折线。
2. 多策略规划：
   - 最短距离：Dijkstra。
   - 最短时间：边权 = 距离/速度（速度可按拥挤度调整）。
   - 多交通工具：边包含交通方式属性。
3. 多点路径规划（TSP）：
   - 小规模（建议 <10）使用状态压缩 DP 或回溯。
4. 室内导航：
   - 节点带 floor。
   - 电梯/楼梯作为跨楼层特殊边。

### 模块三：个性化旅游推荐（智能体现）
目标：实现可解释推荐与高效 Top-K 产出。

关键点：
1. 多维度评分：热度、评价、兴趣标签联合打分。
2. 用户与 POI 标签系统：支持向量匹配（如余弦相似度）或规则加权。
3. Top-K 算法：手写 Max Heap 或固定大小 Min Heap，复杂度 O(N log K)。

### 模块四：旅游日记管理与交流（内容特色）
目标：形成可记录、可检索、可压缩的内容能力。

关键点：
1. 日记管理：文本、多媒体链接、时间与地点关联。
2. 全文检索：手写倒排索引（关键词 -> 日记 ID 列表）。
3. 无损压缩：手写 Huffman 编码/解码流程。
4. AIGC 扩展：可调用第三方 API 生成短视频链接（选做增强）。

### 模块五：AI 智能体应用（贯穿全局）
目标：在开发流程与成果呈现中体现 AI 协作能力。

关键点：
1. 开发辅助：任务拆解、文档生产、演示素材生成。
2. 报告分析：明确 AI 的优势、局限与人工兜底场景。

## 4. 优先级、阶段计划与里程碑

### 4.1 开发优先级
1. 第一优先：模块二（路线规划）+ 模块一（查询管理）。
2. 第二优先：模块三（个性化推荐）。
3. 第三优先：模块四（日记与压缩）。
4. 持续贯穿：模块五（AI 工具与复盘）。

### 4.2 三周里程碑
1. 第 1 周（骨架与数据结构）：
   - 完成后端骨架与数据库模型。
   - 完成 Graph/Trie/Heap 初版。
   - 打通最小链路（查询 + 路径 API 可调用）。
2. 第 2 周（核心功能联调）：
   - 完成多策略路径规划。
   - 完成推荐接口与用户偏好闭环。
   - 前后端联调 MVP。
3. 第 3 周（扩展与交付）：
   - 完成倒排索引与 Huffman。
   - 完成测试、性能记录与文档答辩材料。

## 5. 团队分工与协作机制

### 5.1 角色分工（三人）
1. 成员 A（算法负责人）：
   - 负责 Graph/Trie/Heap/TSP/Huffman 的实现与优化。
   - 输出复杂度分析与测试样例。
2. 成员 B（后端负责人）：
   - 负责 FastAPI 路由、Service 层、DB 模型与 CRUD。
   - 维护接口文档与异常处理规范。
3. 成员 C（前端与联调负责人）：
   - 负责 Vue 页面、地图可视化、接口接入。
   - 负责联调与演示链路稳定性。

### 5.2 协作规范
1. Git feature 分支开发 + PR 评审合并。
2. 每周冻结一次接口字段，避免联调抖动。
3. 每次功能提交至少包含一个最小测试样例。

## 6. 开发环境与运行方式

### 6.1 软件环境
1. Python 3.10+
2. Node.js 18+
3. FastAPI + Uvicorn
4. SQLAlchemy + Pydantic
5. Vue 3 + Vite + Axios

### 6.2 工具环境
1. VS Code
2. Git + GitHub
3. FastAPI Swagger（/docs）

### 6.3 本地运行（后端）
1. 创建虚拟环境并安装依赖。
2. 执行 `uvicorn app.main:app --reload --port 8001`。
3. 访问 `http://127.0.0.1:8001/docs` 调试接口。

## 7. 验收标准（周报与答辩统一口径）

每个模块每周至少提交以下内容：
1. 功能演示：截图或录屏。
2. 算法说明：流程、数据结构与复杂度。
3. 指标结果：查询耗时、路径计算耗时、推荐性能等。
4. 风险状态：阻塞项、影响、应对动作。
5. 下周计划：明确到文件和接口。

总体验收关键项：
1. 核心算法手写且可运行。
2. 路径规划与推荐达到可演示状态。
3. 文档、代码、演示三者一致。

---

## 8. 风险清单与应对策略

1. 算法实现复杂，影响联调进度。
   - 应对：先完成可运行版本，再做性能优化。
2. 接口频繁变更，前端反复返工。
   - 应对：按周冻结 schema，变更走评审。
3. 并行开发冲突导致分支污染。
   - 应对：小步提交、及时 rebase、PR 必审。
4. 数据规模上来后查询性能下降。
   - 应对：Trie + Top-K 堆 + 索引优化联合治理。

# 三人协作流程（TravelAgentSystem）

## 1. 目标
建立一个轻量但稳定的协作机制，确保 3 人并行开发时：
1. 主分支始终可用。
2. 改动可追踪、可回滚。
3. 冲突可控，沟通成本低。

## 2. 角色分工
1. 负责人（Maintainer）
- 负责需求拆分、里程碑安排、PR 最终合并。
- 负责保护分支规则与发布节奏。

2. 开发同学 A
- 负责核心功能开发（例如算法或核心服务）。

3. 开发同学 B
- 负责接口联调、前端对接、测试补充或文档更新。

说明：角色可轮换，但每个迭代必须有 1 名合并负责人。

## 3. 分支策略
1. `main`
- 只保留可发布代码。
- 禁止直接 push，必须通过 PR 合并。

2. `feature/<issue-id>-<short-name>`
- 一个任务一个分支。
- 例：`feature/12-route-planner`。

3. `hotfix/<short-name>`
- 线上紧急修复。
- 修复完成后需回合并到 `main`（如有开发分支，也要同步回合并）。

## 4. 开发与提交流程
1. 创建任务
- 每个需求先建 Issue，写明目标、验收标准、负责人。

2. 开发前同步
- 从最新 `main` 拉取代码后再建分支。

3. 本地开发
- 小步提交，提交信息清晰描述目的。
- 推荐提交格式：`type(scope): summary`。

4. 发起 PR
- PR 模板至少包含：
  - 改动内容
  - 验证方式
  - 风险点
  - 是否影响接口/数据库

5. 代码评审
- 至少 1 人通过 Review。
- 所有检查通过后再合并。

6. 合并策略
- 优先使用 Squash Merge，保持主分支历史清晰。

## 5. 质量门禁（最低要求）
1. 本地自检通过后再提 PR：
- 关键路径可运行。
- 基础测试可通过。
- 无明显格式或静态检查错误。

2. PR 必须满足：
- CI 通过（如已配置）。
- Review 通过。
- 无未解决冲突。

## 6. 冲突处理规则
1. 谁提交改动谁优先处理冲突。
2. 先同步 `main`，再解决冲突，再完整回归关键功能。
3. 冲突超过 30 分钟无法解决，立即拉相关同学一起处理。

## 7. 发布节奏
1. 建议固定发布窗口（例如每周 1 到 2 次）。
2. 发布前仅接收低风险改动。
3. 线上问题走 `hotfix/*`，修复后补充复盘记录。

## 8. 日常协作节奏
每天 10 到 15 分钟站会，固定三件事：
1. 昨天完成了什么。
2. 今天计划做什么。
3. 当前阻塞与需要的协助。

## 9. GitHub 仓库建议设置
1. 开启 `main` 分支保护。
2. 设置至少 1 个必需 Review。
3. 开启合并前必须通过状态检查（如 CI）。
4. 限制直接 push 到 `main`。

## 10. 快速执行清单
1. 所有需求先建 Issue。
2. 从 `main` 新建 `feature/*`。
3. 提交代码并发 PR。
4. 至少 1 人 Review 通过。
5. 检查通过后合并到 `main`。
6. 按发布节奏上线，异常走 `hotfix/*`。
